## 1. Кто писал код и как

Весь фронтенд писался через **Claude Code** под управлением промпт-инженера (не фронтендера
по образованию). Workflow был такой: один экран/срез за промпт → Claude Code реализует →
ручная проверка на устройстве → следующий промпт. Промпты намеренно не готовились заранее
пачкой — на бэкенде так делали и часть промптов устаревала к моменту использования.

Практическое следствие: код **консистентен по паттернам** (каждый новый экран явно
ссылался на предыдущий как образец), но **не проходил полноценный код-ревью фронтендером** —
только typecheck/lint от Claude Code + функциональная проверка на устройстве человеком.
Если вы фронтендер — это первое, на что стоит посмотреть свежим глазом.

## 2. Архитектура — коротко

Монорепо, pnpm workspaces, без turborepo.

```
cargo-frontend/
  packages/
    tokens/      # @cargo/tokens — дизайн-токены, TS, framework-agnostic, hex-цвета
    api/         # @cargo/api — типизированный API-клиент, генерация типов из OpenAPI,
                 #              error-коды, БЕЗ React/TanStack — чистый транспорт
    ui/          # @cargo/ui — веб-компоненты (web-ops), React DOM + Tailwind, Storybook
  apps/
    mobile/      # Expo (React Native, SDK 56) — CLIENT + KG_WORKER в одном бинаре
    web-ops/     # Vite + React — JP_WORKER / KG_WORKER(склад) / ADMIN, ещё не начат
```

`apps/pvz` (отдельный PWA для ПВЗ) **удалён** — был частью раннего плана (ТЗ v1.0/v1.1),
упразднён в v1.2 в пользу нативного стека внутри `apps/mobile`. Код был на другой платформе
(React DOM, не RN) — переиспользовать напрямую было нельзя, поэтому просто снесли, а не
мигрировали. Если ищете референс flow выдачи — смотрите ТЗ §7, не git history `apps/pvz`
(хотя он там есть, если интересна история решений).

**Два ролевых стека в одном мобильном бинаре** — ключевое архитектурное решение, не трогать
без веской причины:
- Роль приходит **с сервера** (`GET /me`), гейтится в `RootNavigator.tsx`.
- Нет UI-переключателя «режим клиент / сотрудник» — у пользователя одна роль.
- `role=CLIENT` → `ClientStack`. `role=KG_WORKER` → `PvzStack` (сейчас заглушка).
- Это два почти не пересекающихся navigation-дерева. Общее — только auth/токены/
  дизайн-система/API-клиент из `packages/*`.

## 3. Что реализовано и проверено на устройстве

**Шаги 0–3** (монорепо, дизайн-токены, API-слой, базовые компоненты) — готовы.

**Шаг 4 — клиентский стек мобилки — закрыт полностью.** Все пять экранов прошли
функциональную проверку на физическом Android-устройстве (EAS dev build), не только
typecheck.

### 3.1 Авторизация (эталонный паттерн)
`PhoneScreen.tsx` → `CodeScreen.tsx`. Телефон → OTP по SMS → токены в SecureStore.
`CodeScreen.tsx` — самый богатый паттерн в кодбейсе (RHF + zod, мутация с лоадером,
серверный countdown через route-param, обработка ошибок через `errorToMessage`,
i18n). Если нужен образец «как тут принято писать формы» — смотреть сюда.

Тестовый клиент: телефон, оканчивающийся на `0001`, роль `CLIENT`, room `C1003`.
OTP-коды в dev достаются из Postgres `notifications` (порт 5433) — SMS-адаптер
стаб, реальные SMS не уходят.

### 3.2 Роль-гейт
`RootNavigator.tsx`: `useMe()` вызывается **только** в ветке `authenticated` (иначе
бил бы 401 на бутстрапе). Loading → Skeleton, error → сообщение + Retry + Выйти,
далее по `me.data.role`: `CLIENT` → `ClientStack`, `KG_WORKER` → `PvzStack`
(заглушка), другая роль → inline «Нет доступа».

### 3.3 Клиентские экраны (ClientStack.tsx)

| Route | Экран | Что делает |
|---|---|---|
| `Address` | `AddressScreen.tsx` | Стартовый экран. Адрес склада из `useMe().jpAddress` — это **одна строка**, не объект (расхождение с ранним ТЗ), парсится на клиенте по разделителю `" / "`. Каждая часть — копируемая строка (Clipboard + Toast). |
| `ParcelList` | `ParcelListScreen.tsx` | Список своих посылок, `useParcels()`. Бэкенд уже фильтрует CLAIMED + сортирует по `createdAt desc`, **paged** (`PageResponse.items`, не голый массив). Pull-to-refresh. |
| `ParcelDetail` | `ParcelDetailScreen.tsx` | Таймлайн **только по оси `status`** (`history`, не `parcel_status_history` — реальное имя поля). `claim_status` — отдельный `ClaimBadge`, не шаг таймлайна. `assigned_pvz=null` → нейтральный текст «будет назначен», не ошибка. |
| `TrackSearch` | `TrackSearchScreen.tsx` | Поиск `GET /me/parcels/search?number=` (**под токеном**, не публичный — расхождение с ТЗ v1.1/v1.2, исправлено в v1.3). Состояния A/B/C/D по `claimable`/`claim_status`. Claim — **один тап**, пустое тело (форма-декларация была в ранней версии claim, бэкенд её убрал). Dispute — `BottomSheet` с опциональным комментарием, `409 ALREADY_DISPUTED` / `422 CANNOT_DISPUTE` — отдельные сообщения. |
| `PreAlertList` | `PreAlertListScreen.tsx` | Список/создание/мягкое удаление. Статусы реальные — `ACTIVE/MATCHED/CANCELLED` (не `PENDING/RECEIVED` из раннего ТЗ). DELETE — мягкий (`204` → `CANCELLED`, запись остаётся в БД, экран фильтрует). Категории — `GET /api/v1/categories` (см. §5, было захардкожено, потом бэкенд добавил эндпоинт, фронт переключили). |
| `Profile` | `ProfileScreen.tsx` | Данные из `useMe()` (без отдельного запроса) — телефон, `maskedRoom` (camelCase по факту, не `masked_room`). Выбор предпочтительного ПВЗ через `BottomSheet` + `GET /pvz` + `PATCH /me/preferred-pvz`. Кнопка «Выйти» — локальный logout (`tokenStore.clear()`). |

### 3.4 UI-компоненты (`apps/mobile/src/ui/`)
Не пакет `@cargo/ui` (тот веб-only) — отдельный набор для RN, построен на `@cargo/tokens`.
Экспортируются через барель `../ui`: `Button`, `Input`, `Card`, `BottomSheet`, `Toast`/
`useToast`, `Skeleton`, `EmptyState`, `StatusBadge`, `ClaimBadge`, `Timeline`, плюс
лейблы (`STATUS_LABELS`, `CLAIM_LABELS`). Все обкатаны в бою на реальных экранах
(не просто лежат в `Showcase.tsx`).

### 3.5 i18n
`apps/mobile/src/i18n/`: `ru.ts` — источник истины (dot-namespaced ключи, `TKey`
выводится автоматически, `t()` типобезопасен), `ky.ts` — пустой `DeepPartial<typeof ru>`
с фолбэком на ru по каждому ключу. JP вне скоупа.

Namespaces сейчас: `common, phone, code, authed, nav, address, pvz, parcels, parcel,
track, prealert, profile, noAccess, errors, validation`.

(`parcels.*` — список, `parcel.*` — деталь, это два разных namespace, не опечатка.)

## 5. Известные риски и технический долг

**Закрыто (раньше было открыто, упомянуто для истории — не тратьте время повторно):**
- ~~Захардкоженные категории pre-alert~~ — был временный хардкод, пока у бэкенда не
  было `GET /api/v1/categories`. Эндпоинт появился, фронт переключён на
  `useCategories()`, хардкод убран. Если увидите упоминание в старых коммитах —
  это историческое, не актуальное состояние.
- ~~Неполный OpenAPI-артефакт~~ — `packages/api/src/generated/schema.ts` отставал
  от живого бэкенда несколько раз за время разработки (не содержал актуальных
  путей `/me/parcels/...`). Перегенерирован, актуален на момент написания. **Если
  бэкенд продолжает меняться — перегенерируйте перед стартом новой работы**
  (`pnpm --filter @cargo/api gen:types` или аналогичная команда — проверить
  `package.json` пакета).

**Открыто:**
- `PvzStack.tsx` — заглушка, не реализация. Это весь объём шага 6.
- Контракт API **менялся трижды только за время шага 4** (track-поиск:
  публичный → под токеном → другое имя query-параметра; claim: с формой-декларацией
  → без неё; pre-alert категории: хардкод → эндпоинт). **Не доверяйте письменному
  ТЗ буквально — всегда сверяйтесь с живым `/api-docs` перед написанием кода**,
  это было главным источником трения на протяжении всего шага 4 и почти наверняка
  повторится на шагах 5–7.
- Read-only ограничение разработчика (предыдущего) — тестовые данные не подчищались
  напрямую в БД, только через API. Если в dev-окружении остались тестовые записи
  (например, `CANCELLED` pre-alert на тестовом клиенте) — это не баг, это следы
  ручного тестирования.

## 6. Дизайн-система

- Источник — `packages/tokens` (TS-объект, framework-agnostic, hex-цвета,
  unitless числа).
- Палитра: нейтральная база + акцент **вермильон/киноварь** (`#D93A2B`),
  опасность (danger) — **отдельный холодный карминовый**, намеренно отделён
  от акцента (чтобы «акцентная» кнопка не читалась как «опасное действие»).
- Статусные цвета семантические: зелёный — `DONE`, янтарный — `UNCLAIMED`,
  красный — `DISPUTED`/`LOST`. Полная палитра status/claim — в `StatusBadge`/
  `ClaimBadge` + `STATUS_LABELS`/`CLAIM_LABELS` в `ui/labels`.
- Web (`packages/ui`) — Tailwind 4 CSS-first (`@theme`), плотность через
  `data-density` атрибут на `<html>`. Mobile — токены напрямую в `StyleSheet`,
  плотность не варьируется (мобильный тач-таргет минимум 44px везде).

## 7. Если продолжать по плану (рекомендуемый порядок)

1. **Шаг 6** (ПВЗ-стек мобилки) — логично сразу после шага 4, RN-контекст
   свежий. Workflow прямо это рекомендует (см. `CARGO_Frontend_Workflow.md`,
   примечание к шагу 6).
2. **Шаг 5** (Web Operations: приём + консолидация) — отдельный стек (Vite+React),
   с нуля, можно начинать параллельно с шагом 6 если есть второй человек.
3. **Шаг 7** (Web Operations: хаб + админ) — после шага 5, тот же стек.
4. **Шаг 8** (доводка) — по мере, не блокирует демо ничего из вышеперечисленного.

Если решите **не продолжать по плану** (переписать, сменить стек и т.п.) —
сохраните как минимум: дизайн-токены (`packages/tokens`), понимание двухосевой
модели статусов (ТЗ §3) и архитектуру «роль с сервера, без UI-переключателя»
(ТЗ §2) — это продуктовые решения, не вкусовщина реализации, их придётся
переоткрывать заново, если выбросить вместе с кодом.
