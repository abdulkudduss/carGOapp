# CARGO — Frontend Workflow (план работ по шагам)

> **Для кого:** фронтенд-разработчик (или ты в его роли). Каждый шаг имеет «готово, когда» — не переходить дальше, пока критерий не выполнен. Дизайн и код экранов генерируем через Claude; промпты для каждого шага готовятся по мере прохождения (как делали с бэкендом).
> **Стек (зафиксировано):** мобилка — React Native (Expo), два ролевых стека в одном бинаре (CLIENT / KG_WORKER-ПВЗ); Web Operations — React (Vite); общие дизайн-токены. PWA для ПВЗ упразднён (ненадёжность камеры/пушей на iOS) — ПВЗ-флоу нативный внутри мобилки.

---

## Шаг 0 — Окружение и доступы (полдня)

1. Поднять бэкенд локально: клон репозитория, `docker compose up -d`, `./gradlew bootRun` (нужен JDK 21 — см. README/требования сборки). Открыть Swagger UI, авторизоваться.
2. Прочитать `CARGO_Frontend_TZ_v1.1.md` целиком — особенно раздел 3 (две оси) и раздел 12 (чего не делать).
3. Через Swagger создать тестовые данные на каждую роль: клиент (OTP-логин), JP_WORKER, KG_WORKER с ПВЗ, ADMIN; принять пару посылок, одну заклеймить. Сохранить телефоны/токены в заметку.
4. Создать монорепо (pnpm workspaces):
```
cargo-frontend/
  packages/tokens/      # дизайн-токены (TS) — единственный источник
  packages/api/         # типизированный API-клиент (общий)
  packages/ui/          # веб-компоненты (Web Operations)
  apps/mobile/          # Expo (React Native) — ДВА ролевых стека: CLIENT (ЛК) + KG_WORKER (ПВЗ-выдача)
  apps/web-ops/         # Vite + React — админ/Япония/хаб
```
Библиотеки: TanStack Query (данные/кэш), react-hook-form + zod (формы), zustand (лёгкий стейт), Tailwind на вебе (токены → CSS-переменные), StyleSheet/токены в RN.
> **Примечание (v1.2):** если скелет уже содержит `apps/pvz` (от раннего PWA-плана) — он упраздняется, его flow переезжает в `apps/mobile` как ПВЗ-стек. Удалить пакет и ссылки на него.

**Готово, когда:** бэкенд отвечает локально, тестовые пользователи созданы, монорепо собирается двумя пустыми приложениями (mobile + web-ops).

## Шаг 1 — Дизайн-токены (через Claude, 1 сессия)

Сгенерировать `packages/tokens`: палитра (нейтральная база + один акцент; **семантические цвета статусов под обе оси** — 10 значений `status` + 3 значения `claim_status` из раздела 3 фронтенд-ТЗ), типографика (2 размера шкалы: компактная для Web Ops, крупная для mobile/PVZ), отступы, радиусы, тени. Выход — TS-объект + генерация CSS-переменных для веба.

**Готово, когда:** токены лежат в `packages/tokens`, подключены в оба приложения, есть страница-витрина (все цвета/шрифты на одном экране).

---

## Шаг 2 — API-слой (1 день)

1. Сгенерировать типы из OpenAPI бэкенда (`openapi-typescript` по Swagger JSON) → `packages/api`.
2. Обёртка fetch/axios: подстановка `Authorization: Bearer`, перехват `401` → refresh → ретрай → логаут; хелпер `Idempotency-Key` (UUID на операцию).
3. Обработка ошибок: парсинг машинных кодов (`ALREADY_CLAIMED`, `CONFIRMATION_REQUIRED`, …) в типизированный union — UI маппит код, а не HTTP-статус.

**Готово, когда:** из консоли тестового приложения проходит полный цикл: OTP-запрос → verify → `GET /me` → `401`-сценарий с refresh.

---

## Шаг 3 — Базовые компоненты (через Claude)

Два набора на одних токенах:
- `packages/ui` (веб): Button, Input (+вариант «сканер-поле» с автофокусом), Badge (status/claim — единственное место, где задаётся их вид), Card, Table, Timeline, Sheet/Modal, Toast, Skeleton, EmptyState.
- `apps/mobile/src/ui` (RN): Button, Input, Badge, Card, Timeline, BottomSheet, Toast, Skeleton.

**Готово, когда:** Storybook (веб) / витринный экран (RN) показывают все компоненты; Badge корректно отображает все 13 статусных значений.

## Шаг 4 — Mobile App, клиентский стек: ядро (главный демо-милстоун)

Сначала — **ролевой гейт на входе** (`/me` → выбор стека CLIENT / KG_WORKER, см. фронтенд-ТЗ раздел 2). Эталонный экран OTP-логина это уже доказал; здесь гейт оформляется как точка ветвления на два navigation-стека (второй — ПВЗ — пока заглушка, наполняется на шаге 6).

Затем клиентские экраны в порядке ценности:
1. OTP-логин (таймер повтора — с сервера).
2. «Мой адрес» — карточка с построчным копированием (`GET /me`).
3. Список посылок → карточка с таймлайном по `parcel_status_history`; состояние «ПВЗ будет назначен» для `assigned_pvz = null`.
4. Поиск по треку → claim → обработка `409 ALREADY_CLAIMED` → спор (`409/422` различать, см. ТЗ 6.4).
5. Pre-alerts (список/создание/удаление, `409` на дубль).
6. Профиль: выбор ПВЗ (`GET /pvz` → `PATCH /me/preferred-pvz`).

**Готово, когда:** полный клиентский путь проходит на реальном бэкенде: логин → увидел адрес → нашёл UNCLAIMED по треку → заклеймил → посылка в списке с таймлайном. **Это первое демо заказчику.**

## Шаг 5 — Web Operations: приём + консолидация (операционное ядро)

1. Каркас: логин, ролевой роутинг, layout с навигацией.
2. Экран приёма (ТЗ 5.1A): цикл «скан трека → вес/категория → создать → печать label.pdf → фокус назад» — **полностью без мыши**, Idempotency-Key на каждый сабмит. Печать = открыть PDF → `window.print()`.
3. Консолидация: создание бокса, режим наполнения (скан-поле, счётчик, повторный скан = спокойный no-op), извлечение, seal с весом, печать этикетки бокса.
4. Список посылок с фильтрами.

**Готово, когда:** на реальном бэкенде принимается посылка и собирается бокс без касания мыши; этикетки печатаются.

---

## Шаг 6 — Mobile App, ПВЗ-стек (выдача, роль `KG_WORKER`)

> **Порядок:** этот стек живёт в той же мобилке, что шаг 4. Удобно делать его **сразу после шага 4** (RN-контекст и дизайн-система свежие), не дожидаясь веба шагов 5/7 — нумерация оставлена прежней, но шаги 5 и 6 независимы и порядок между ними на твоё усмотрение.

Наполняем второй navigation-стек (заглушку из шага 4) реальными экранами. Гейт уже выбирает его по роли `KG_WORKER` + `home_warehouse_id`.

1. Нативная камера-сканер (Expo Camera / vision-camera + декодер QR) + ручной ввод кода как фолбэк.
2. Поиск клиента (`/issuance/clients?phone=&code=`) → посылки к выдаче → DRAFT.
3. Flow подтверждения: OTP request/confirm → skip с причиной (пресеты + текст) → подпись (нативный canvas → PNG ≤200KB) → complete с обработкой `409 CONFIRMATION_REQUIRED` и `INVALID_STATUS`.
4. Cancel на любом шаге.

**Готово, когда:** полная выдача проходит на реальном устройстве с камерой (минимум один прогон на железе — эмулятор камеру не покрывает).

---

## Шаг 7 — Web Operations: хаб + админ

1. Хаб: приём партии (скан бокса → массовый AT_KG, сверка «заявлено/факт»), сортировка по ПВЗ (назначение `assigned_pvz` посылкам без него).
2. Админ: споры (список DISPUTED → resolve с тремя исходами), справочник точек, **редактирование адреса с подтверждением рассылки**, тарифы/курс (просмотр).

**Готово, когда:** спор, созданный с мобилки на шаге 4, разрешается из админки; смена адреса проходит подтверждение.

---

## Шаг 8 — Доводка (по мере, не блокирует демо)

- **Offline-outbox приёма JP** (IndexedDB-очередь + индикатор) — самая сложная фронтенд-задача, делается осознанно отдельно, когда онлайн-приём уже обкатан.
- i18n: все строки через словарь с шага 1 (ru заполняем, ky — позже).
- Push-уведомления (Expo notifications) — когда бэкенд-notification доведён.
- Пустые состояния/скелетоны/ретраи — добиваются по каждому экрану.

---

## Сквозные правила работы

- Каждый шаг = ветка → демо на реальном бэкенде → мердж. Никаких моков, бэкенд уже живой.
- Экраны генерим через Claude **после** того, как готовы токены и компоненты шагов 1–3 — и всегда со ссылкой на них в промпте.
- Любое «бэкенд так не отвечает» — фиксируется и сверяется со Swagger, а не обходится на фронте.
- **Дистрибуция мобилки (решить до релиза):** одно приложение в сторе на обе роли (сотрудник логинится рабочим номером) — дефолт. Если нужен отдельный «рабочий» дистрибутив — два EAS-профиля из одного кода. Открытый вопрос к фронтендеру, не блокирует разработку.
