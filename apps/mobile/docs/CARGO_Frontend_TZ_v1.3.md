# CARGO (NBM JAPAN) — Frontend ТЗ v1.3

> **Для кого:** фронтенд-разработчик. Документ самодостаточен — читать бэкенд-ТЗ не обязательно.
> **Конвенция:** технические идентификаторы (статусы, поля, роуты) — на английском, пояснения — на русском. В коде и API используем только английские идентификаторы.
> **Источник истины по API:** Swagger UI бэкенда (`/swagger-ui.html`). Этот документ описывает «что и зачем», Swagger — «как именно».

---

## 1. Что за продукт (2 минуты на понимание)

Карго-компания доставляет посылки из Японии в Кыргызстан. Клиент покупает в японских интернет-магазинах и указывает адресом доставки **склад компании в Японии** + свой персональный **Room-номер** (например `336-1045`). Склад принимает посылку, взвешивает, клеит свою QR-этикетку, упаковывает в боксы, боксы едут партией в КР. В КР посылка попадает в пункт выдачи (ПВЗ), клиент платит и забирает.

Ключевая особенность: посылка может приехать на склад **без понимания, чья она** (на коробке только трек-номер магазина). Система не блокирует такие посылки — они едут дальше как «невостребованные», а клиент позже находит их по трек-номеру и «привязывает» к себе.

## 2. Карта приложений — что делаем

| # | Приложение | Платформа | Пользователи | Суть |
|---|---|---|---|---|
| 1 | **Web Operations** | Desktop web (responsive не критичен) | `ADMIN`, `JP_WORKER`, `KG_WORKER` (главный склад) | Рабочее место: таблицы, сканер, печать этикеток, массовые операции |
| 2 | **Mobile App** | Мобильное приложение (iOS/Android, React Native) | `CLIENT` **и** `KG_WORKER` (сотрудник ПВЗ) | Один бинарь, два независимых интерфейса по роли: ЛК клиента **или** рабочее место выдачи ПВЗ |

Все приложения ходят в **один и тот же REST API**. Никакой логики «для мобилки свой бэкенд» нет. Что показывать пользователю — определяется его ролью (роль приходит в JWT и в `/me`).

**Почему ПВЗ — нативный экран в мобилке, а не отдельная PWA (решение v1.2):** на iOS PWA — гражданин второго сорта (капризная камера в standalone, ненадёжные пуши, неочевидная установка). Сотруднику ПВЗ камера-сканер критична ежедневно, поэтому ПВЗ-флоу живёт нативными экранами внутри той же React Native мобилки и получает полноценные нативные камеру/canvas. Это **чисто фронтовое решение — бэкенд не меняется**, API ролевой.

**Архитектура «двух ролей в одном бинаре» (обязательное условие):**
- Роль приходит **с сервера** (`GET /me`), интерфейс гейтится по ней. **Никакого UI-переключателя «режим клиент / режим сотрудник»** — у пользователя ровно одна роль, выбирать нечего.
- `role=CLIENT` → монтируется клиентский navigation-стек (ЛК). `role=KG_WORKER` + `home_warehouse_id` → монтируется стек выдачи ПВЗ.
- Это **два почти не пересекающихся приложения внутри одного бинаря.** Общее у них только: auth, токены, дизайн-система, API-клиент (всё из `packages/*`). Держать как два отдельных navigation-стека, выбираемых по роли на входе — НЕ сливать в одно дерево экранов с общим стейтом.

Сканеры: в Web Operations используются аппаратные USB/Bluetooth-сканеры — для фронта это просто **быстрый ввод с клавиатуры в сфокусированное поле + Enter**. В мобилке (ПВЗ-стек) сканируем нативной камерой (Expo Camera / vision-camera + декодер QR).

## 3. Главная ментальная модель — ДВЕ независимые оси

Это самое важное место документа. У посылки **два независимых статуса**, и их нельзя смешивать в голове и в UI:

**Ось 1 — `status` (где посылка физически):**

| `status` | Лейбл RU (предложение) | Комментарий |
|---|---|---|
| `AT_JP` | На складе в Японии | |
| `PACKED` | Упакована к отправке | Лежит в боксе |
| `SHIPPED` | В пути | |
| `CUSTOMS` | На таможне | Post-MVP, в UI заложить место |
| `AT_KG` | Прибыла в Кыргызстан | |
| `DELIVERY` | Передана в доставку/ПВЗ | |
| `DONE` | Выдана | Терминальный |
| `RETURNED` | Возвращена | Терминальный |
| `LOST` | Утеряна | Терминальный |
| `DISPOSED` | Утилизирована | Терминальный |

**Ось 2 — `claim_status` (чья посылка):**

| `claim_status` | Лейбл RU | UI |
|---|---|---|
| `UNCLAIMED` | Не привязана | Серый/жёлтый бейдж. У посылки `client_id = null` |
| `CLAIMED` | Привязана к клиенту | Обычное состояние, бейдж можно не показывать клиенту |
| `DISPUTED` | Спорная | Красный бейдж, виден только `ADMIN` |

**Следствия для UI:**
- Посылка может быть `(SHIPPED, UNCLAIMED)` — едет, но ничья. Это нормально, не баг.
- Таймлайн посылки рисуем **только по оси `status`**. `claim_status` — отдельный бейдж рядом, не шаг таймлайна.
- Клиент видит только свои (`CLAIMED` на него) посылки. `UNCLAIMED`-посылки клиент находит через **поиск по треку под токеном** (раздел 6.4).
- Никогда не вычисляйте статус на клиенте. Текущий `status`, `claim_status` и история переходов всегда приходят с сервера.

| Роль | Приложение | Кто это |
|---|---|---|
| `CLIENT` | Mobile App | Клиент компании |
| `JP_WORKER` | Web Operations | Сотрудник склада в Японии |

## 5. Приложение 1 — Web Operations

### 5.1 Экраны `JP_WORKER` (склад Японии)

**A. Приём посылки (главный экран, оптимизировать под скорость):**
1. Поле фокуса: скан/ввод оригинального трек-номера (`track_number`). Кнопка «Трек не читается» — приём без трека.
2. Система ищет pre-alert по треку. Нашла → показывает клиента, посылка сразу станет `CLAIMED`. Не нашла → создаётся `UNCLAIMED` (это норма, не ошибка — не пугать оператора красным).
3. Ввод веса (кг, десятичные), выбор категории (`categories` с сервера), опционально фото (upload).
4. Подтверждение → бэкенд создаёт parcel, возвращает `parcel_code` → **печать внутренней этикетки**: бэкенд отдаёт PDF этикетки, фронт открывает его в окне печати браузера (термопринтер выбран принтером по умолчанию). Никакой интеграции с принтером на фронте — только `window.print()` / печать PDF.
5. Курсор автоматически возвращается в поле трека — следующая посылка. Весь цикл должен проходить без мыши (Tab/Enter).

**B. Консолидация (упаковка в боксы):**
1. Кнопка «Создать бокс» → бэкенд генерирует `box_code` → печать этикетки бокса.
2. Режим «наполнение»: сканируем внутренние QR посылок одну за другой → посылка добавляется в бокс, счётчик растёт, статус посылки → `PACKED`. Список содержимого на экране, возможность убрать посылку из бокса.
3. «Запечатать бокс» (`sealed`) → ввод веса бокса.

**C. Список посылок на складе:** таблица, фильтры по `status`/`claim_status`/дате, поиск по `parcel_code`/`track_number`.

**Offline-режим (важно):** приём посылок должен работать при падении интернета. Операции складываются в локальную очередь (IndexedDB) и синхронизируются при восстановлении. Каждая операция отправляется с заголовком **`Idempotency-Key`** (UUID, генерируется на клиенте при создании операции, при ретраях не меняется) — бэкенд гарантирует, что дубль не создастся. В UI — индикатор «N операций в очереди».

### 5.2 Экраны `KG_WORKER` (главный склад КР)

**A. Приём партии:** скан `box_code` → бэкенд массово переводит все посылки бокса в `AT_KG` → на экране список содержимого бокса с подсветкой расхождений (заявлено N, факт сканов M — для пересчёта вручную при подозрении).

**B. Сортировка по ПВЗ:** скан посылки/бокса → система показывает целевой `pickup_point` (или даёт выбрать) → статус `DELIVERY`. *Логика распределения по ПВЗ упрощённая в MVP — экран делаем, правила доopределим.*

### 5.3 Экраны `ADMIN`

- **Дашборд:** счётчики по статусам, посылки в пути, невостребованные старше N дней.
- **Споры (`DISPUTED`):** список конфликтных привязок, карточка спора (две стороны, доказательства), решение: подтвердить одного / отвязать всех.
- **Пользователи и роли**, **тарифы**, **курс JPY→KGS** (просмотр, история).
- **Адрес склада в Японии — редактируемый.** При сохранении нового адреса бэкенд рассылает push/SMS всем активным клиентам. В UI — подтверждение «Адрес изменится у всех клиентов, отправить уведомления?».
- Поиск любой посылки + ручные операции (смена статуса с комментарием, ручная привязка).

## 6. Приложение 2 — Client Mobile App

### 6.1 Авторизация
Только телефон + OTP по SMS. Без паролей. Flow: ввод номера → `POST /auth/otp/request` → экран ввода кода (+ таймер повторной отправки, его длительность отдаёт сервер) → `POST /auth/otp/verify` → пара токенов (см. раздел 9).

### 6.2 Главный экран — «Мой адрес в Японии»
Самая используемая фича. Карточка с адресом склада, где в нужное место подставлен персональный Room клиента (`masked_room`, например `336-1045`). **Каждая строка адреса — отдельная кнопка «копировать»** (клиент вставляет построчно в форму японского магазина). Адрес приходит с сервера (`GET /me/address`) и **никогда не хардкодится** — админ может его сменить в любой момент.

### 6.3 Мои посылки
Список (карточки: трек/код, лейбл статуса, вес, ориентировочная сумма). Карточка посылки: **вертикальный таймлайн** по `parcel_status_history` (шаг = переход `status`, с датой), вес, фото, сумма к оплате (приходит рассчитанной с сервера — **на клиенте ничего не считаем**, курс фиксируется бэкендом).

### 6.4 Поиск по треку + Claim (привязка) — ключевой сценарий
Клиент купил товар, но не сделал pre-alert. Посылка приехала как `UNCLAIMED`.

1. Экран «Найти посылку» (доступен **залогиненному** клиенту): ввод трек-номера → `GET /api/v1/me/parcels/search?number=...` (**под токеном**, Bearer). Ответ: `parcel_id` (int64), `status`, `claim_status`, `location` (текст), `claimable` (true только для UNCLAIMED), `message`. Вес, фото, имя владельца — не отдаются. Анонимный запрос → `401`. «Не найдено» → код `PARCEL_NOT_FOUND`.
2. Если `claimable=true` → кнопка «Это моя посылка» → `POST /api/v1/parcels/{id}/claim` (`id` = `parcel_id` из ответа поиска). Тело **пустое** (или опц. `{comment}`) — личность из JWT, форма-декларация НЕ нужна. Если у клиента в профиле задан предпочтительный ПВЗ — он наследуется в посылку; если нет — посылка остаётся без ПВЗ (это нормально, см. 6.7).
3. **`409 ALREADY_CLAIMED`** на claim: посылку уже привязал другой клиент. Сообщение: «Эту посылку уже привязал другой клиент. Если вы уверены, что она ваша — откройте спор» + кнопка «Оспорить».
4. **Спор:** `POST /api/v1/parcels/{id}/dispute` (опциональный комментарий) → посылка `DISPUTED`, экран «Спор передан администратору». Коды ошибок различать:
   - `409 ALREADY_DISPUTED` — этот клиент уже оспаривал эту посылку → «Ваш спор уже на рассмотрении»;
   - `422 CANNOT_DISPUTE` — посылка своя или UNCLAIMED, спор не имеет смысла → не показывать кнопку в этих состояниях вовсе (а код обработать как защиту).
5. Если посылка уже `CLAIMED` другим при поиске (`claimable=false`, `claim_status=CLAIMED`) — сразу сценарий спора, без кнопки claim.

### 6.5 Pre-alert (опционально, но есть в MVP)
Форма: трек-номер, магазин, категория, ориентировочная стоимость. Список своих pre-alert'ов со статусом «ждём посылку / получена».

### 6.6 Прочее
Уведомления (push + история в приложении; уведомления приходят **только** по своим, т.е. `CLAIMED`, посылкам), профиль, экран «как покупать» (инструкция с адресом).

### 6.7 Выбор пункта выдачи (ПВЗ)
- `GET /api/v1/pvz` — публичный список активных ПВЗ (`id`, `name`, `address`).
- В профиле клиент может выбрать предпочтительный ПВЗ: `PATCH /api/v1/me/preferred-pvz` body `{ warehouse_id }`. Выбранный ПВЗ наследуется всеми последующими claim'ами.
- **У посылки ПВЗ может отсутствовать** (`assigned_pvz` = null) — клиент не выбирал, назначат при сортировке в КР. В карточке посылки это штатное состояние: показывать «Пункт выдачи будет назначен по прибытии», не пустую строку и не ошибку.

## 7. Mobile App — ПВЗ-стек (рабочее место выдачи, роль `KG_WORKER`)

Нативный интерфейс сотрудника ПВЗ **внутри той же мобилки** (выбирается по роли из `/me`, см. раздел 2). Та же OTP-авторизация, что у клиента. Большие кнопки, минимум текста — работа стоя, одной рукой.

### 7.1 Поиск клиента / посылок
Скан внутреннего QR посылки камерой **или** поиск клиента: `GET /api/v1/issuance/clients?phone=&code=` → выбор клиента → `GET /api/v1/issuance/clients/{clientId}/parcels` — посылки, доступные к выдаче (статусы `AT_KG`/`DELIVERY`, привязанные). Групповая выдача — клиент обычно забирает несколько.

### 7.2 Flow выдачи (главный сценарий)
Выдача на бэкенде — сущность со статусами `DRAFT → COMPLETED` (или `CANCELLED`). Фронт ведёт пользователя по шагам:

1. Отметили чекбоксами посылки → `POST /api/v1/issuance` body `{ client_id, parcel_ids[] }` + заголовок `Idempotency-Key` → создан DRAFT. Ответ содержит позиции и **итоговую сумму** (рассчитана сервером с фиксацией курса — на клиенте не пересчитывать).
2. Оплата наличными в MVP — подтверждается на шаге complete (онлайн-оплата O!Bank — post-MVP, место в UI заложить).
3. **Подтверждение получения — двухступенчатое:**
   - **OTP (основной):** `POST /api/v1/issuance/{id}/otp/request` → клиенту SMS → сотрудник вводит код со слов клиента → `POST /api/v1/issuance/{id}/otp/confirm` body `{ code }`. Неверный код → `400` (есть лимит попыток — показывать остаток не нужно, просто сообщение).
   - **Деградация:** SMS не доходит → `POST /api/v1/issuance/{id}/otp/skip` body `{ reason }` (причина обязательна, выбор из пресетов + свободный текст). После skip подпись обязательна.
   - **Подпись:** canvas на весь экран → PNG → `POST /api/v1/issuance/{id}/signature` (multipart, ≤200KB). Кнопки «очистить» и «готово». При подтверждённом OTP подпись опциональна (но клиент-заказчик хочет её всегда — собираем по умолчанию).
4. `POST /api/v1/issuance/{id}/complete` body `{ payment_method: "CASH" }` → посылки `DONE`, экран «Выдано ✓». **`409 CONFIRMATION_REQUIRED`** — не выполнено условие «OTP подтверждён ЛИБО (skip + подпись)»: вернуть пользователя на шаг подтверждения, подсветив, чего не хватает. Повторный complete (двойной тап, ретрай сети) безопасен — вернётся `200`.
5. «Отменить» на любом шаге до complete → `POST /api/v1/issuance/{id}/cancel` (посылки не затрагиваются).

Редкий кейс: complete может вернуть ошибку `INVALID_STATUS`, если посылку параллельно выдали в другой выдаче. Показать «Посылка уже выдана, обновите список» и перезагрузить состав.

### 7.3 Технические требования (нативные)
- Камера-сканер QR — нативный (Expo Camera / react-native-vision-camera + декодер), фолбэк — ручной ввод кода.
- Подпись: нативный canvas (`react-native-signature-canvas` или SVG/skia-canvas) → PNG, целиться в ≤200KB.
- ПВЗ-стек работает онлайн (офлайн-очередь для выдачи в MVP не нужна, в отличие от склада Японии на вебе).
- ПВЗ-стек и клиентский стек делят auth/токены/API-клиент/дизайн-систему, но это **раздельные navigation-стеки** — не общий граф экранов (см. раздел 2).

## 8. Сквозные правила для всех приложений

1. **Сервер — источник истины.** Статусы, суммы, курсы, таймеры OTP — всё приходит с бэкенда. Фронт ничего не вычисляет и не «предсказывает».
2. **`409`/`422` — это не ошибки системы, а бизнес-ответы** (конфликт claim, бокс запечатан, подтверждение не пройдено). У каждого в теле есть машинный код (`ALREADY_CLAIMED`, `CONFIRMATION_REQUIRED`, …) — маппить код на конкретное сообщение/действие, не показывать тост «Ошибка 409».
3. **Мутации, которые могут ретраиться** (офлайн-очередь склада, плохая сеть в ПВЗ), отправляются с `Idempotency-Key`.
4. **Денежные суммы** отображаем как пришли (сумма + валюта + зафиксированный курс, если бэкенд его отдаёт). Никаких умножений на клиенте.
5. Пустые состояния и ошибки сети — обязательные состояния каждого экрана (скелетоны, retry).
6. Языки: RU — основной, KG — заложить i18n с первого дня (все строки через словарь), JP не нужен.

## 9. Авторизация и работа с API

- **JWT:** `access` (живёт 15 минут) + `refresh`. Access — в память/secure storage, не в localStorage на вебе (обсудим httpOnly cookie vs header — финализирует бэкенд, ориентируйтесь на Swagger).
- Перехватчик на `401`: попытка `refresh` → ретрай запроса → при провале refresh — логаут на экран входа.
- Все запросы с `Authorization: Bearer <access>`.
- OTP-логин одинаков для всех ролей; роль и контекст (включая `pickup_point` для сотрудника ПВЗ) — из `GET /me`.
- Swagger UI уже работает (identity-модуль готов), авторизация в Swagger через bearer. Эндпоинты остальных модулей появляются по мере готовности — синхронизируемся по мере выкатки.

## 11. Глоссарий

| Термин | Значение |
|---|---|
| **Track number** | Внешний трек японского перевозчика/магазина. Используется для поиска посылки клиентом (под токеном) и claim. На складе после приёма не используется |
| **Parcel code** | Наш внутренний уникальный код посылки, формат `CARGO-{8 символов}` (например `CARGO-A1B2C3D4`). Печатается в QR на нашей этикетке. Все операции внутри системы — по нему |
| **Box / box code** | Транспортная коробка с посылками, код формата `BOX-{ггММ}-{NNNN}` (например `BOX-2606-0042`). Скан бокса = массовая операция над содержимым |
| **Batch** | Партия (рейс) из боксов. Иерархия: Batch → Box → Parcel |
| **Pre-alert** | Заявка клиента «жду посылку с таким треком» до её прихода. Даёт автопривязку при приёме |
| **Claim** | Привязка `UNCLAIMED`-посылки к клиенту |
| **Masked room** | Персональный Room-номер клиента в адресе склада (например `336-1045`) |
| **PVZ / пункт выдачи** | Точка выдачи в КР. Технически — `warehouse` типа `KG_PVZ`. Список: `GET /api/v1/pvz`. У посылки ПВЗ может быть не назначен (null) — штатное состояние |
| **Issuance** | Запись о выдаче: посылки, сумма, оплата, OTP-подтверждение, подпись |

## 12. Чего НЕ делать (типичные ошибки)

- ❌ Не объединять `status` и `claim_status` в один список статусов / один таймлайн.
- ❌ Не считать суммы, не конвертировать валюту, не вычислять «следующий статус» на клиенте.
- ❌ Не хардкодить адрес склада, тарифы, длительности таймеров.
- ❌ Не показывать оператору `UNCLAIMED` как ошибку — это штатный режим приёма.
- ❌ Не сканировать оригинальные штрихкоды магазинов во внутренних операциях — после приёма работаем только с нашим QR (`parcel_code`).
- ❌ Не блокировать UI приёма на складе при потере сети — очередь + индикатор.
